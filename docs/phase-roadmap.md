# Phase Roadmap

## Scope and operating principle

This roadmap keeps the first phase documentation-led and evidence-first. We avoid full feature implementation until covenant feasibility is demonstrated.

## Phase 0 — Scaffold and assumptions (now)

- [x] Create explicit repo scaffold and constraint statements.
- [x] Document why roulette and web app are deferred.
- [x] Publish environment check script.
- [x] Define terminology and feasibility framing.

## Phase 1 — TN12 minimal covenant spike planning

- [ ] Confirm minimal successful path and acceptance criteria.
- [ ] Record required tooling, accounts, and network endpoints.
- [ ] Capture command examples in spike docs.
- [ ] Mark every step as verified / unverified.

## Phase 2 — Execute and record the spike

- [ ] Attempt tiny covenant artefact creation on TN12/testnet.
- [ ] Attempt spend/closure of that artefact.
- [ ] Capture inspection outputs and reproducible interpretation steps.
- [ ] Fill in findings with explicit limitations and caveats.

## Phase 3 — Readiness review

- [ ] Consolidate verified evidence into feasibility doc.
- [ ] Re-run environment checks and note any version-sensitive behavior.
- [ ] Decide one of:
  - **Go**: enough evidence to define roulette PoC constraints.
  - **Pause**: blockages require protocol/toolchain changes.
  - **No-go**: current tooling gaps are too high for fair-game PoC.

## Exit criteria for this repo phase

- No full technical implementation has been added.
- Documents consistently distinguish between knowns, unknowns, assumptions, and verified results.
- Environment check remains non-invasive and safe to run repeatedly.
