3) Signing route

What is now source-proven about official kaspa-wallet:
- cli/src/modules/pskb.rs exposes these wallet commands:
  - pskb create <address> <amount> <priority fee>
  - pskb sign <pskb>
  - pskb send <pskb>
  - pskb debug <payload>
  - pskb parse <payload>
  - pskb script lock <payload> <amount> [priority fee]
  - pskb script unlock <payload> <fee>
  - pskb script sign <pskb>
  - pskb script address <pskb>
- wallet/core/src/account/mod.rs implements:
  - pskb_from_send_generator(...)
  - pskb_sign(...)
  - pskb_broadcast(...)
- wallet/core/src/account/pskb.rs shows PSKB signing uses locally derived account private keys in memory and does not require exporting secrets into public artifacts.

What is also source-proven about wallet data structures:
- wallet/core/src/tx/payment.rs defines PaymentOutput with an optional covenant field.
- wallet/pskt/src/output.rs and wallet/pskt/src/pskt.rs carry covenant data in PSKT outputs.

Critical limitation found:
- wallet/core/src/tx/generator/generator.rs currently maps PaymentDestination::PaymentOutputs to:
  TransactionOutput::new(output.amount, pay_to_address_script(&output.address), None/implicit no covenant path)
  effectively dropping PaymentOutput.covenant during the normal generator path.
- So the CLI send/create path is not a proven covenant-output authoring route even though the underlying types know about covenant fields.

Meaning for signing:
- Official kaspa-wallet can sign PSKB bundles.
- Official kaspa-wallet can sign ordinary account-owned inputs and the dedicated P2SH script flow.
- Official kaspa-wallet is not source-proven here as a complete covenant create/spend shell workflow, because the exposed CLI create path has no covenant-binding arguments and the normal generator path drops covenant data.
- `pskb script sign` appears aimed at P2SH payload unlocks, not at building SilverScript covenant-declaration sigscripts.
