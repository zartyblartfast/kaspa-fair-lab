2. Is there a separate kaspad flag such as --mine, --enable-mining, --mining, or similar?
- No built-in mine/start-mining flag was found in kaspad help.
- The only mining-related kaspad flag exposed here is --enable-unsynced-mining.
- Local source/help meaning: this affects whether RPC block submission is accepted while the node is unsynced; it is not a command to start mining.

3. Is --utxoindex required for mining or only for later UTXO lookup?
- Local evidence indicates --utxoindex is not required for mining itself.
- Mining RPC path uses get_block_template and submit_block in rpc/service/src/service.rs and does not gate those methods on config.utxoindex.
- UTXO/balance RPC methods do gate on config.utxoindex and return NoUtxoIndex otherwise.
- README evidence also says utxoindex is needed for wallet functionality.
- Recommendation: not required for a first mining-only attempt; required later if you want local address/UTXO lookup via node RPC.

4. Can --utxoindex be enabled on the existing synced data without resync/reindex?
- Restart is required to change the flag.
- Local startup code in kaspad/src/daemon.rs shows enabling --utxoindex creates a separate utxoindex data directory if requested.
- The same startup code does not show a mandatory full DB delete just because utxoindex is toggled on.
- However, the exact one-time cost of populating the UTXO index from the existing consensus state is not fully documented in the local materials reviewed here.
- Conservative answer: full consensus resync is not clearly required by the code reviewed, but some index build/catch-up work should be expected after restart. Treat exact duration/cost as UNKNOWN until explicitly tested.

- docs/testnet12.md says it is very important not to forget --netsuffix=12, otherwise the node will connect to mainnet or the default testnet.
- kaspad/src/args.rs shows default testnet suffix is 10.
- Therefore --testnet alone is not sufficient for TN12.
